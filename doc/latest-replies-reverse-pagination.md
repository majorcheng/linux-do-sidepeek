# 首帖 + 最新回复（逆序分页）方案

> 说明：文档中较早章节提到的“最新回复优先”是旧叫法；当前产品名称已收敛为“首帖 + 最新回复”。


## 1. 背景

当前抽屉预览中的 `最新回复优先` 实现，本质上是对当前已拿到的 `post_stream.posts` 做局部倒序：

- 首帖固定在顶部
- 其余 replies 执行 `reverse()`

这个实现只在 **当前拿到的是完整 post_stream** 时才可靠。

对于 Discourse 长帖，主题 JSON 往往只返回一个局部窗口，而不是完整回复流，因此会出现：

- UI 显示“最新回复优先”
- 实际只是“当前加载窗口里的回复倒序”
- 并不代表整帖真正最新的回复

这会导致功能语义与用户预期不一致。

---

## 2. 目标

将当前的 `最新回复优先` 升级为更准确、可扩展的模式：

- 首帖固定在顶部
- 首次打开时优先展示“真正最新的一批回复”
- 用户继续向下滚动时，按时间倒序继续加载“更早的一批回复”
- 不要求一次加载完整长帖
- 保持抽屉预览轻量，不退化为完整阅读器

---

## 3. 推荐产品定义

建议将当前 `最新回复优先` 的语义收敛为：

- `默认顺序`
- `仅首帖`
- `首帖 + 最新回复`

如果暂时不改 UI 文案，也应在内部按这个语义实现，而不是承诺“完整主题倒序浏览”。

### 推荐展示方式

抽屉中的内容顺序如下：

1. 首帖
2. 最新一批 replies（从新到旧）
3. 用户继续向下滚动后，加载更早的一批 replies
4. 再往下滚动，继续加载更早的 replies

即视觉顺序为：

```text
首帖
最新回复
次新回复
更早回复
更更早回复
...
```

这是一种 **逆序分页**，不是“完整主题全量倒序”。

---

## 4. 为什么选择逆序分页

### 4.1 优势

- **语义更真实**：展示的是“最新回复流”
- **长帖可用**：不需要一次性拉完整主题
- **性能更好**：避免大主题全量请求和全量渲染
- **更适合抽屉预览**：保持轻量、快速、局部阅读的定位

### 4.2 不建议的替代方案

#### A. 仅对当前窗口做 `reverse()`

问题：
- 长帖中并不能代表真正最新回复
- 容易误导用户

#### B. 一次性拉完整主题后再倒序

问题：
- 请求重
- 渲染慢
- 对长帖体验差
- 违背“预览”本身应轻量的目标

#### C. 通过 iframe / 整页 DOM 滚动抓取内容

问题：
- 难以稳定控制
- 不易去重
- 和整页模式耦合太深
- 不适合作为智能预览的数据来源

---

## 5. 核心实现思路

### 5.1 数据来源

继续使用 Discourse 的 JSON / topic post 数据接口。

不要依赖：

- iframe 内页面滚动
- 整页 HTML 提取
- DOM 抓取

### 5.2 初次加载

当用户开启“最新回复”模式并打开主题时：

1. 获取主题基础数据（title、meta、首帖、统计信息等）
2. 如果主题较短，且已拿到完整 `post_stream.posts`
   - 可直接构建“首帖 + replies 倒序”视图
3. 如果主题较长，`post_stream` 不完整
   - 额外请求“尾部窗口”数据
   - 获取最后一批 replies
   - 构建“首帖 + 最新一批 replies”视图

### 5.3 下滑继续加载

当用户滚动到回复列表底部附近时：

1. 找到当前已加载 replies 中“最旧的一条 reply”
2. 以该 reply 为边界，请求更早的一批 replies
3. 将新批次去重后，追加到当前列表底部
4. 更新“是否还有更早回复”的状态

注意：

- 视觉上是“向下滚动查看更多较早回复”
- 时间顺序上是“从新到旧逐步展开”

---

## 6. 数据模型调整建议

当前实现把 `topic.post_stream.posts` 直接当成最终渲染数据，这不适合倒序分页。

建议拆分为以下状态：

### 6.1 建议新增状态

- `firstPost`
  - 首帖对象，始终固定显示在顶部
- `replyMap`
  - `Map<post_number, post>`，用于去重和增量合并
- `replyOrder`
  - 当前 replies 的展示顺序，按“从新到旧”维护
- `oldestLoadedPostNumber`
  - 当前已加载 replies 中最旧的一条，用于继续向前翻页
- `hasMoreOlderReplies`
  - 是否还有更早的 replies 可加载
- `isLoadingOlderReplies`
  - 是否正在加载更早回复，防止重复请求
- `latestRepliesInitialized`
  - 是否已完成“最新回复模式”的初始窗口加载

### 6.2 为什么需要 `Map`

因为分页窗口之间可能出现重叠，必须按 `post_number` 去重。

当前的 `mergeTopicPreviewData()` 已有合并思路，但更适合升级为：

- 统一维护 `replyMap`
- 每次请求新批次后增量合并
- 再根据 `replyOrder` 输出到视图层

---

## 7. 渲染模型建议

### 7.1 视图结构

建议把主题视图拆成两部分：

1. `first post section`
2. `replies stream section`

这样可以确保：

- 首帖不会因为分页或排序逻辑被混入 replies
- replies 可以独立增量更新
- 后续若加“仅首帖 + 最新 N 条”也更容易扩展

### 7.2 用户提示文案

在最新回复模式下，建议显示提示：

- 当前为“最新回复流”视图
- 首帖固定在顶部
- 向下滚动可查看更多较早回复
- 如需完整阅读或原始顺序，请打开原帖

如果当前只加载了部分回复，还应提示：

- 当前仅显示最新一部分回复，不代表完整主题全部内容

## 8. 与现有代码的对应关系

当前重点相关位置：

- `src/content.js:650` `getVisiblePosts()`
- `src/content.js:861` `shouldFetchTargetedTopic()`
- `src/content.js:877` `topicHasCompletePostStream()`
- `src/content.js:912` `mergeTopicPreviewData()`
- `src/content.js:567` `renderTopic()`
- `src/content.js:586` `buildTopicView()`

### 8.1 建议的重构方向

#### `getVisiblePosts()`
当前职责：
- 根据设置从 `posts` 中截取/倒序

建议改为：
- 普通模式：继续处理完整/当前窗口 posts
- 最新回复模式：改为从新的 replies state 中取数据
- 不再直接依赖一次性 `posts.reverse()`

#### `renderTopic()`
建议增加：
- 初始化最新回复模式的数据状态
- 判断是否需要额外请求尾部窗口
- 根据不同模式选择不同渲染分支

#### `mergeTopicPreviewData()`
建议保留思路，但不要再把它作为唯一的数据聚合方式。
更适合拆成：

- `mergeReplyBatch(replyMap, batch)`
- `buildReverseReplyOrder(replyMap)`
- `appendOlderReplyBatch(...)`

## 9. 分页触发建议

### 9.1 触发方式

建议监听抽屉内容区域滚动：

- 当滚动接近底部阈值时触发加载
- 例如距离底部 `300px` 内时开始请求下一批

### 9.2 防抖与并发保护

必须保证：

- 同一时间只有一个“加载更早回复”的请求在进行
- 请求完成前，不重复发起下一次
- 关闭抽屉或切换主题时，旧请求可中止或被丢弃

### 9.3 加载状态

建议在 replies 区域底部显示：

- `正在加载更早回复...`
- `已显示全部可加载回复`
- `加载失败，点击重试`

---

## 10. 深链接与目标语义

当前项目已经支持：

- `/t/<slug>/<id>/<post_number>`
- `/t/<slug>/<id>/last`

在最新回复模式下，需要注意：

### 10.1 `/last`
非常适合作为“最新回复模式”的初始目标语义。

### 10.2 指定楼层 `/123`
如果用户点击的是指定楼层链接，其语义优先级通常高于“最新回复流”。

建议规则：

- 如果点击的是明确楼层定位链接，优先保留楼层目标语义
- 不要强行改成“最新回复流起点”
- 最新回复模式更适合作用于普通主题入口或 `/last` 语义入口

---

## 11. 风险与边界

### 11.1 风险：尾部窗口接口行为依赖 Discourse 实际返回
需要先确认：

- `/last.json` 或等价 JSON 请求是否稳定返回最后窗口
- 返回的 posts 是否足以构造“最新一批 replies”
- 是否存在窗口大小限制或特殊主题类型差异

### 11.2 风险：重复计数 / 额外请求
当前扩展预览会尽量兼容浏览计数。
倒序分页新增请求后，需要确认：

- 后续分页请求不应意外重复触发浏览计数逻辑
- 统计相关 header 只在需要时发送
- 加载更早 replies 不应等价成多次“访问主题”

### 11.3 风险：性能与内存
对于极长主题：

- 即使是倒序分页，也可能不断堆积 reply 节点
- 需要考虑最多保留多少条已渲染回复
- 后续若需要，可再引入简单的分段卸载或虚拟列表

当前阶段可先不做，但文档中需要保留该风险说明。

---

## 12. 建议的最小可行版本（MVP）

第一阶段建议先做一个可控版本：

### MVP 范围

1. 保留首帖固定显示
2. 最新回复模式下，初次加载“最后一批 replies”
3. 支持向下滚动继续加载更早一批 replies
4. 批次间按 `post_number` 去重
5. 加载状态和结束状态可视化
6. 明确提示“当前为最新回复流，不代表完整主题原始顺序”

### MVP 暂不处理

- 虚拟列表
- 超大主题分段卸载
- 极复杂的定位与最新流混合策略
- 对所有类型深链接做完美兼容

---

## 13. 回归测试建议

实现后至少回归以下场景：

### 13.1 短主题

- 最新回复模式打开后，排序正确
- 不应额外请求不必要的分页数据

### 13.2 长主题

- 首次打开应显示首帖 + 最新一批 replies
- 向下滚动应继续加载更早回复
- 不应出现重复楼层
- 不应打乱首帖位置

### 13.3 `/last` 链接

- 应优先符合“最新位置”的用户语义
- 抽屉初始视图应合理

### 13.4 指定楼层链接

- 不应被错误改造成纯最新流
- 楼层定位语义应优先保留

### 13.5 切换主题 / 关闭抽屉

- 旧请求不应污染新主题视图
- 状态应正确重置

### 13.6 浏览计数

- 初次打开预览的计数行为符合当前设计
- 分页请求不应造成异常重复计数

---

## 14. 最终结论

当前的“最新回复优先”在长帖中不可靠，问题不在于排序本身，而在于数据源不是完整回复流。

推荐的修复方向不是继续对当前窗口做 `reverse()`，而是将该功能升级为：

- **首帖固定 + 最新 replies 逆序分页**

这套方案能同时满足：

- 功能语义更准确
- 长帖体验更好
- 性能更可控
- 与抽屉预览的轻量定位一致

如果后续落地实现，建议优先按 MVP 推进，再根据真实使用反馈决定是否继续扩展为更完整的回复流阅读模式。

## 15. 当前评估结论（2026-03-08）

结合当前仓库实现与 Discourse 路由行为，结论如下：

- 这份方案方向是对的，问题判断准确
- 当前 `最新回复优先` 在长帖里确实不可靠
- 原因不是排序代码本身，而是当前拿到的数据通常只是局部窗口，不是完整回复流

### 15.1 可行性判断

- **MVP：高可行性**
- **完整逆序分页：中等可行性**

原因：

- Discourse 现有 JSON 路由可以支持该方向
  - `/t/<slug>/<topic_id>/last.json` 可用于拿尾部窗口
  - `/t/<slug>/<topic_id>/<post_number>.json` 可用于拿指定楼层附近窗口
  - `/t/<topic_id>/posts.json` 可用于后续按批次补更早回复
- 不需要依赖 iframe DOM 抓取，也不需要一次性加载完整主题

### 15.2 当前实现的限制

当前代码仍属于“静态窗口渲染”模型：

- `renderTopic()` 基本按一次请求结果整块重渲染
- `getVisiblePosts()` 只是对当前 posts 做切片或 `reverse()`
- 尚未维护独立的 replies stream state

这意味着：

- 短帖中 `最新回复优先` 可能看起来没问题
- 长帖中它只是“当前窗口倒序”，不能承诺是“整帖真正最新回复优先”

### 15.3 推荐开发策略

不建议一步做到完整版逆序分页。

推荐按两阶段推进：

#### 第一阶段（推荐先做）

- 保持智能预览默认仍为 **正常顺序**
- 将“倒序”功能明确为：**首帖 + 最新回复流**
- 长帖下首次打开时，优先请求尾部窗口并展示“最新一批 replies”
- 暂不做滚动继续加载更早回复

这样可以先解决“长帖语义错误”的主要问题，同时避免一次引入过多状态和回归风险。

#### 第二阶段（可后续再做）

- 增加向下滚动时继续加载更早回复
- 引入 `replyMap`、`oldestLoadedPostNumber`、`hasMoreOlderReplies` 等状态
- 将该模式完整升级为逆序分页回复流

### 15.4 产品定义建议

建议将用户可感知的语义写清楚，不再暗示“完整主题全量倒序”。

推荐口径：

- 默认：**正常顺序**
- 仅首帖：**只看首帖**
- 倒序模式：**首帖 + 最新回复**

其中“倒序模式”需要明确说明：

- 首帖固定在顶部
- 下面显示最新一批回复，并按从新到旧排列
- 长帖下默认不是把整帖所有回复一次性倒过来
- 如需完整阅读或原始顺序，应打开原帖

### 15.5 为什么这样更合适

- 与当前抽屉“轻量预览”定位一致
- 对长帖更诚实，不会误导用户
- 改造成本可控
- 能减少对深链接、计数、整页模式等现有逻辑的冲击
