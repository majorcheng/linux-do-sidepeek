# Linux.do / Discourse 规则记录

这份文档用于记录在开发 `Linux.do SidePeek` 过程中，已经确认或需要持续遵守的 `linux.do` / Discourse 行为规则。

目标不是介绍功能，而是给后续开发、调试、回归测试提供一个统一参考。

## 1. 站点基础事实

- `linux.do` 基于 Discourse。
- 主题链接的常见路径形式为：
  - `/t/<slug>/<topic_id>`
  - `/t/<slug>/<topic_id>/<post_number>`
  - 短格式 `/t/<topic_id>`
  - 短格式楼层定位 `/t/<topic_id>/<post_number>`
- 主题详情、列表、搜索、用户流、分类、标签页都可能出现主题链接，但这些链接的语义并不完全一样。

## 2. 浏览计数规则

### 2.1 已观察到的行为

- 仅仅请求主题地址，并不一定会计入浏览。
- 用户“左键点击标题进入主题”更接近会触发浏览计数的真实路径。
- “右键 + 新标签打开”这类方式，可能不会计入浏览。
- 隐藏 iframe、后台上下文、非正常前台路由进入，不能假设一定会计数。

### 2.2 对扩展实现的含义

- 如果要尽量兼容浏览计数，不能只依赖简单的页面请求。
- 更接近 Discourse 原站行为的方式是：
  - 请求主题 JSON 时带 `track_visit=true`
  - 同时带上 `Discourse-Track-View`
  - 同时带上 `Discourse-Track-View-Topic-Id`
- 是否最终计数，仍然由站点后端规则决定，扩展只能“尽量贴近”。

### 2.3 推荐测试方法

- 在列表中找一个相对小众、当前浏览数比较稳定的帖子。
- 先记下列表里的浏览数字。
- 用左键点击标题，测试智能预览或整页模式。
- 停留几秒后返回列表并刷新，观察浏览数是否增加。
- 为避免误判，尽量避开热门帖子，减少其他用户同时点击造成的数字跳动。

## 3. 主题链接接管规则

### 3.1 允许接管的链接

- 应优先接管“主题主标题链接”。
- 可支持的常见页面包括：
  - 首页、`latest`、`new`、`unread`、`top`、`hot`
  - 分类页、标签页
  - 搜索结果页
  - 用户发帖流、收藏等带主题标题链接的页面

### 3.2 不应误接管的链接

以下链接即使也是 `/t/...`，也不能简单一律接管：

- 标准列表里的“最后活动时间 / 最新回复”链接
  - 用户点它时，通常是想跳到某一楼或未读位置
  - 不能把它错误改成“打开抽屉并从首帖开始看”
- 主题正文中的链接
- 编辑器中的链接
- 各类弹层、菜单、用户卡片、群组卡片中的链接

### 3.3 对实现的要求

- “是否接管”与“链接是否属于主题”不是一回事。
- 需要区分：
  - 主标题链接
  - 时间戳 / 最新回复 / 未读定位链接
  - 正文中的普通主题链接
- 对于标准 topic list，不能因为链接是 `/t/...` 就全部拦截。

## 4. 深链接与楼层定位规则

### 4.1 深链接的意义

Discourse 经常使用这种链接：

- `/t/<slug>/<topic_id>/<post_number>`
- `/t/<slug>/<topic_id>/last`

这些链接常见于：

- 未读主题
- 搜索命中
- 最新活动跳转
- 用户流 / 活动流中的定位链接

它们的含义不是“打开这个主题”，而是：

- 打开这个主题
- 并尽量定位到指定楼层 / 未读位置 / 最后位置

### 4.2 开发规则

- 不能把用户点击的目标路径无脑压平成主题根路径。
- 以下行为都应尽量保留原始目标：
  - 抽屉打开的目标 URL
  - `整页模式` 的 iframe URL
  - `新标签打开` 的 href
- 如果智能预览使用 JSON 渲染：
  - 需要尽量根据点击目标保留目标语义，而不只是数字楼层
  - 如果链接是数字楼层，应滚到对应回复
  - 如果链接是 `/last` 这类 helper target，应尽量滚到对应位置

### 4.3 一个重要原则

要区分两类 URL：

- **用户点击目标 URL**
  - 用来保留用户真实意图
  - 例如 `/t/foo/123/80`
- **主题 canonical key / topic id**
  - 用于计数、去重、prev/next 导航等内部逻辑

这两者不能混为一谈。

## 5. Topic Path 解析规则

### 5.1 纯数字第二段优先按短格式回复链接解释

对于 Discourse 当前路由，下面这种纯数字路径：

- `/t/2024/123`

应优先理解为：

- `topic_id = 2024`
- `post_number = 123`

而不是把 `2024` 当成纯数字 slug。

如果 slug 中包含数字但并非纯数字，例如：

- `/t/123-reasons/456`

这时才应按 slugged topic path 处理。

### 5.2 解析顺序要求

更稳妥的顺序应是：

1. 如果 `/t/` 后第一段是纯数字，按短格式 `/t/<topic_id>(/<post_number>)?` 处理
2. 否则，如果第二段是纯数字，按 `/t/<slug>/<topic_id>(/<post_number>)?` 处理

否则会造成：

- 请求错误的 `.json`
- 发送错误的计数 topic id
- 楼层定位错误
- 智能预览错误回退或打开错误主题

## 6. Prev / Next 导航规则

- `上一帖 / 下一帖` 以及 `Alt + ↑ / Alt + ↓` 应该按“当前页面可见条目”切换，而不是按页面里所有 `/t/...` 链接切换。
- 标准列表中一行可能同时存在：
  - 主标题链接
  - 最新回复 / 时间戳链接
- 搜索结果、用户流里，同一主题也可能以多条可见记录分别出现。

### 开发要求

- 标准列表中，同一行里的多个主题链接只能算一个导航条目
- 搜索结果、用户流里，不同可见条目即使属于同一主题，也应分别保留
- 因此，prev/next 不能简单按完整 href 去重，也不能全局按 topic id 去重

## 7. 智能预览与整页模式的边界

### 智能预览

- 优先走 JSON 渲染，加载更轻。
- 与原帖页不保证 100% 一致。
- 对长帖、复杂嵌入、动态组件、部分定位场景，需要接受一定差异。

### 整页模式

- 更接近原帖页真实表现。
- 适合需要完整展示效果的情况。
- 如果用户点击的是带楼层定位的目标，iframe 也应尽量保留这个目标路径。

## 8. 回归测试建议

每次涉及链接识别、计数、导航、楼层定位时，至少回归以下场景：

### 8.1 列表点击

- `latest` 页面点主标题，抽屉正常打开
- `latest` 页面点“最后活动 / 最新回复”链接，不应被错误接管
- 分类页、标签页主标题点击正常

### 8.2 搜索 / 用户流

- 搜索结果中的主题标题能正常打开抽屉
- 用户发帖流中的主题标题能正常打开抽屉

### 8.3 楼层定位

- 点击 `/t/<slug>/<id>/<post_number>` 时：
  - `新标签打开` 保留目标
  - `整页模式` 保留目标
  - 智能预览若能拿到对应楼层，应尽量滚到目标位置

### 8.4 浏览计数

- 左键进入抽屉预览后，列表浏览数应尽量能增长
- 智能预览与整页模式都需要验证
- 右键新标签打开是否计数，不应作为扩展成功标准

### 8.5 特殊路径

- 短格式主题路径：`/t/123`
- 短格式楼层定位：`/t/123/5`
- slugged 主题路径：`/t/foo/123`
- 含数字但非纯数字 slug：`/t/123-reasons/456`

## 9. 当前文档用途

这份文档主要用于：

- 修改 `src/content.js` 时快速对照行为边界
- 回答“为什么这个链接不能直接接管”
- 回答“为什么计数不能只靠 iframe / 后台请求”
- 作为 review 时的规则依据，避免同类回归反复出现
